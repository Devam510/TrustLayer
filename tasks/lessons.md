# Lessons Learned

- [2026-03-29] -- Explicit User Approval -> ALWAYS wait for explicit user approval before executing tasks if the system requires it.
- [2026-03-29] -- Bitnami Docker Image Tagging -> Do NOT use `bitnami/pgbouncer:latest` as `latest` is unreliable. Pin explicit image versions or use alternative images.
- [2026-03-29] -- Localhost vs 127.0.0.1 in Node Postgres -> When local DB migrations fail with 'password authentication failed', ensure no duplicate `.env` files are present in the package directory (`packages/db/.env`) that override the root configuration. Ensure `127.0.0.1` is used instead of `localhost` to avoid Node resolving to IPv6 `::1` which might connect to a native Windows PostgreSQL instance rather than the Docker container.
- [2026-03-29] -- dotenv Not Auto-Loaded in TSX Scripts -> `drizzle-kit` auto-loads `.env` via its config, but raw `tsx src/file.ts` scripts do NOT. Any script using `process.env.*` must explicitly call `dotenv.config({ path: '../../.env' })` at the top before any DB client is instantiated. Pattern: mirror the same dotenv path used in `drizzle.config.ts`.
- [2026-03-29] -- pg_partman is Production-Only -> `pg_partman` is not installed in bare Docker PostgreSQL images. Any seed step depending on it must be wrapped in its own `try/catch` with a graceful warning so local dev seeding doesn't crash. It will succeed correctly on Supabase/managed Postgres.
- [2026-03-29] -- Apostrophe Inside Single-Quoted JS Strings -> A straight ASCII apostrophe (char 39, `'`) inside a single-quoted string terminates the string early, causing parser errors like "Expected ',', got 'll'". This is NOT a curly/smart-quote issue — it's basic JS string delimiter collision. Pattern: any contraction (`you'll`, `don't`, `it's`) inside `'...'` will break. Fix: use double quotes `"..."` or template literals `` `...` `` for any string containing an apostrophe. Previous incorrect diagnosis (curly apostrophe) was wrong — always check raw char codes before concluding.
- [2026-03-29] -- TypeScript Compilation Blocks due to Unused Locals/Parameters -> When refactoring a large partially-typed NestJS codebase, temporarily disable `noUnusedLocals` and `noUnusedParameters` in `tsconfig.json` to prevent superficial lint-style build failures from breaking valid code compilation and deployment.
- [2026-03-29] -- NestJS Startup ENOENT: public.pem -> The API uses RS256 for JWT token verification. If `apps/api/certs/public.pem` or `private.pem` does not exist, the NestJS application will crash on startup rather than during compilation. Generate them inside the `apps/api/certs` directory with OpenSSL or the native node crypto module.
- [2026-03-29] -- Missing pino-pretty -> When NestJS application starts and outputs "Error: unable to determine transport target for 'pino-pretty'", it implies the dependency is missing from the API's package.json. `pnpm add pino-pretty` inside `apps/api` resolves this.
- [2026-03-29] -- Multiple `.env` File Synchronization -> When changing global infrastructure configuration (like a local Postgres port from 5432 to 5434), you MUST ensure ALL `.env` files in the monorepo (e.g. `apps/api/.env`, `apps/web/.env`, `packages/db/.env`) are synchronized. A drifted `apps/api/.env` port caused silent "password authentication failed" internal server 500 crashes internally, which misleadingly surface to Next.js clients as a CORS/network "Failed to fetch" error due to missing API startup initialization syncing.
- [2026-03-29] -- Next.js Dev Port Shift Breaks CORS -> Next.js `dev` server auto-shifts from port 3000 to 3001 (or higher) if port 3000 is already occupied. This silently changes the request `Origin` header and causes CORS rejection showing as "Failed to fetch". Fix: always kill all servers before starting fresh. Make NestJS CORS origin a dynamic array including both ports in dev. Never leave `$env:PORT=XXXX` in a PowerShell session.
- [2026-03-29] -- CSP connect-src Blocks API Fetch -> A Content-Security-Policy `connect-src 'self'` in `next.config.mjs` headers will block ALL `fetch()` calls to any cross-origin URL, including `localhost:4000`. The browser enforces CSP before sending the request, so the error appears as a generic "Failed to fetch" with no network request visible in DevTools Network tab. Fix: add the API origin explicitly to `connect-src` (e.g. `http://localhost:4000` for dev, real domain for production). This is the FIRST thing to check when `fetch()` fails silently with no request appearing in the Network tab.
- [2026-03-30] -- React Element Type Invalid (Named vs Default Exports) -> "Element type is invalid: expected a string... but got: undefined." usually means an export/import mismatch. In Next.js App Router, if a component is defined as `export function TrustIndicator`, it MUST be imported as `import { TrustIndicator } from '...'`. Importing it as a default `import TrustIndicator` will produce `undefined` at runtime and crash the page.
- [2026-03-30] -- Server-Sent Events (SSE) block 401 Unauthorized -> The browser `EventSource` API does NOT support custom HTTP headers (like `Authorization: Bearer <token>`). If you protect an `@Sse()` endpoint with a global `JwtAuthGuard`, the connection silently loops or fails with 401. Fix: You MUST configure `ExtractJwt.fromExtractors([..., ExtractJwt.fromUrlQueryParameter('token')])` in `jwt.strategy.ts` and append `?token=...` to your EventSource URL.
- [2026-03-30] -- Prevent Secrets in Git (`.pem` / `certs/`) -> By default, `git` tracks everything. When scaffolding an auth service that generates RS256 `private.pem` & `public.pem` keys locally, they will push to GitHub unless explicitly ignored. Root Pattern: Missing preemptive `.gitignore` setup. Rule: When generating certificates or keys, IMMEDATELY add `*.pem`, `*.key`, and the `certs/` folder to `.gitignore` before making any commits. To fix retroactively without deleting local files: `git rm -r --cached certs/`. Note: Since the private key was pushed to GitHub, it is COMPROMISED. Never use this specific `private.pem` file in production.
- [2026-03-30] -- GitHub Actions Syntax (Railway & Vercel) -> Third-party GitHub Actions frequently change input syntax (`with:` vs `env:`). Problem: `bervProject/railway-deploy` fails if `railway_token` is passed to `with:`. It explicitly requires `RAILWAY_TOKEN` passed via `env:`. Similarly, older/some versions of Vercel Action do not accept a `scope:` parameter and expect strings for numbers (like `version: '9'`). Rule: Always verify the official action docs for `env` vs `with` when a `.yml` workflow linting fails.
- [2026-04-03] -- NestJS ConfigModule Provider Initialization Timing -> Always use `ConfigService` with `inject: [ConfigService]` for third-party library providers (like Razorpay) rather than directly reading `process.env` inside custom Provider `useFactory`. Trying to read `process.env` directly can crash the backend server instantly because the provider factory may evaluate before NestJS finishes parsing the `.env` file from the disk. When API startup fails contextually like this, it masks itself as a generic "Failed to fetch" on the Next.js frontend!
